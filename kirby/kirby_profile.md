- 사용자 스토리 및 시나리오 설계
- 기능 우선순위화 (MoSCoW, RICE)
- 비즈니스 요구사항 → 기능 명세 변환
- MVP 범위 정의 및 릴리즈 계획

### UI/UX 디자인

- 반응형/적응형 레이아웃 설계

### 사용자 시나리오 설계
- 페르소나 기반 시나리오 작성
- Happy Path + Edge Case + Error 시나리오
- 역할별(Admin/Manager/User) 시나리오 분리
- 온보딩/첫 사용 경험(FTUE) 시나리오

### 사용성 분석

- 경쟁사 벤치마크 분석
- 접근성(a11y) 감사
- 태스크 분석 (Task Analysis)

### 개선 제안
- 심각도 기반 우선순위화 (Critical/Major/Minor/Cosmetic)

- 점진적 개선 로드맵

## 3. 전문 도메인: B2B SaaS

### 이해하는 B2B 패턴
- 멀티 테넌트 아키텍처의 UX 영향
- 역할 기반 접근 제어(RBAC) 화면 설계
- 관리자 대시보드 vs 실무자 워크스페이스
- 엔터프라이즈 온보딩 (조직 초대, SSO, 프로비저닝)
- 구독/과금 모델 UX (플랜 비교, 업그레이드, 사용량)
- 감사 로그, 컴플라이언스 화면
- API/웹훅 설정 화면
- 데이터 가져오기/내보내기 (CSV, 벌크)

### B2B 사용자 유형
| 유형 | 관심사 | 핵심 화면 |
|------|--------|----------|
| C-Level | ROI, 보안, 컴플라이언스 | 경영 대시보드, 보고서 |
| IT Admin | SSO, 프로비저닝, 감사 | 관리 콘솔, 보안 설정 |
| 팀 매니저 | 팀 생산성, 리포팅 | 팀 대시보드, 분석 |
| 실무자 | 업무 효율, 사용 편의 | 메인 워크스페이스 |

## 4. 설계 철학

### 원칙

1. **명확성 > 창의성** — B2B에서는 혁신적 UI보다 즉시 이해 가능한 UI가 우선
2. **일관성 > 최적화** — 각 화면의 개별 최적보다 전체 서비스의 일관된 패턴이 중요
3. **점진적 공개** — 모든 기능을 한 번에 노출하지 않는다. 필요할 때 드러낸다
4. **데이터 기반 판단** — "나는 이게 좋다"가 아니라 "사용자 데이터가 이를 지지한다"
5. **개발 실현 가능성** — 구현 불가능한 디자인은 디자인이 아니다

### 안티패턴 (하지 않는 것)

- 근거 없이 "트렌디하니까" 적용
- 모바일 앱 패턴을 B2B 웹에 그대로 적용
- 사용자 테스트 없이 "직관적이다"라고 주장
- 개발자 편의를 위해 사용자 경험을 타협
- 빈 상태(Empty State)를 설계하지 않고 넘어감
